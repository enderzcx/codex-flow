# Goal Prompt For v1.11

Historical prompt for the planned v1.11 JS Dynamic Runtime MVP.
Use only after `docs/JS_DYNAMIC_WORKFLOWS_PLAN.md` has passed review.

```text
/goal Implement Codex Flow v1.11 JS Dynamic Runtime MVP in /Users/sunny/Work/CODEX/codex-workflows.

Outcome:
- Add a Claude-like JavaScript dynamic workflow runtime for Codex Flow.
- Support preview-first execution for local or generated `workflow.js` harnesses.
- Execute JavaScript through a constrained CWF runtime object plus a concrete containment boundary, not unrestricted Node.js.
- Provide the first safe runtime APIs: `cwf.git`, `cwf.agent.run`, `cwf.map`, `cwf.artifacts`, and `cwf.report`.
- Require explicit approval before a generated or local dynamic workflow executes.
- Preserve existing YAML workflow behavior, v1.7 app-thread behavior, and v1.10 safe-write behavior.
- Produce run artifacts for script source, preview, capabilities, budget, events, agent outputs, and final report.
- Add explicit `cwf.agent.run` permission profiles: default `read-only`, public/auditable `safePatch`, and trusted local `inherit-session` that cannot exceed the parent Codex session permission cap.

Boundaries:

Allowed writes:
- src/ dynamic workflow runtime, CLI command handling, run-store/status/result surfaces, worker adapter integration, and helper modules needed for v1.11.
- package.json/package-lock.json only if a small maintained JavaScript parser dependency is needed for the AST policy gate.
- tests/ dynamic workflow parser/runtime/CLI fixtures and regression coverage.
- fixtures/workflows/ or fixtures/dynamic/ safe local fixture scripts.
- docs/JS_DYNAMIC_WORKFLOWS_PLAN.md, docs/SPEC.md, docs/PHASE_CONTRACTS.md, docs/POST_V1_PLAN.md, README.md, README.zh-CN.md, docs/claude-vs-codex-workflows.md, docs/workflow-catalog.md, and skills/codex-workflows/SKILL.md when behavior/docs must stay aligned.
- GOAL_PROMPT.md and docs/goal-prompts/ for status/handoff updates.

Do not edit:
- Do not add unrestricted `node workflow.js` execution.
- Do not expose `fs`, `child_process`, network, `fetch`, `process.env`, dynamic import, CommonJS `require`, arbitrary package imports, or shell access to workflow scripts.
- Do not rely on Node `vm` alone as the security boundary.
- Do not hand-roll JavaScript parsing with string matching if a parser dependency is needed.
- Do not allow JavaScript itself to write files, read arbitrary target files, spawn processes, access network, or inspect environment directly.
- Do not allow `inherit-session` for remote workflows, public registry workflows, packaged third-party workflows, copied-local workflows, unknown-origin workflows, hash-mismatched workflows, or any workflow not explicitly approved in the initiating local Codex session.
- Do not grant a worker more authority than the parent Codex session already has.
- Do not implement `cwf.write.safePatch()` beyond the already planned public/auditable write profile unless the existing v1.10 safe-write contract is reused.
- Do not enable direct app-thread writes.
- Do not execute remote JavaScript workflows by URL.
- Do not add scheduler, daemon, marketplace, hosted platform, non-Codex model routing, production deploys, databases, credentials, payments, permissions, or external-message writes.
- Do not claim full Claude Dynamic Workflows parity.

Constraints:
- Keep JavaScript as the dynamic orchestration surface.
- Keep all high-impact actions behind CWF APIs and existing safety boundaries.
- Enforce dynamic script containment with a static AST policy gate plus a permissioned Node child process.
- The AST gate must reject imports, dynamic import, require, eval, Function, globalThis, process, constructor/prototype escape paths, direct shell strings, and calls not rooted in `cwf` or explicitly allowed builtins.
- The child process must use Node Permission Model, must not receive target repo filesystem access, network, child process, worker, native addon, WASI, FFI, or inspector permissions, and must communicate with the parent only through CWF JSON-RPC.
- If the local Node runtime cannot support the required permission boundary, dynamic execution must fail closed with a clear unavailable status.
- `cwf.agent.run` must default to `read-only`, support `permissions: "safePatch"` for public/auditable patch-mode writes, and support `permissions: "inherit-session"` only for trusted local workflows.
- `inherit-session` must derive Codex SDK/app-thread sandbox and approval parameters from the parent session, record inherited permission metadata, and refuse to run if the parent permission cap is unknown.
- `inherit-session` must require a trusted workflow origin. v1.11 should allow `generated-current-session` scripts first: CWF generated the script during the initiating Codex session, wrote it into the run directory, recorded its SHA-256, and the approved script hash still matches at execution time.
- `local-trust-record` may be planned for later, but copied local files, remote imports, registry workflows, packaged third-party workflows, unknown origins, and hash mismatches must be rejected before worker execution.
- Workflow origin must be a strict enum: `generated-current-session`, `local-trust-record`, `copied-local`, `remote`, `registry`, `packaged`, or `unknown`. Unknown or unhandled values must fail closed.
- `inherit-session` preview must be non-skippable. When the parent session has broad permissions such as workspace-write or full access, preview must compare declared task scope with inherited capability and highlight surprising or broader-than-needed authority.
- App-thread `inherit-session` fallback must not be silent. If app-thread write inheritance is unavailable, CWF must record `inherit-session-degraded-to-read-only`, show it in status/final summary, and run read-only only after explicit fallback approval.
- Read-only workers must capture target diff before and after execution and fail if the target changes.
- Inherited write workers must record changed files, diff summary, sandbox metadata, and verification evidence in run artifacts and final summary.
- Worker artifacts must be persisted by the parent CWF process into the run folder; after `inherit-session`, CWF must check that no out-of-run artifacts were created beyond declared target changes.
- Generated scripts must be previewed and approved before execution.
- The v1.11 runtime must have budget fuses: max agents, max concurrency, max rounds where loops are supported, wall-clock timeout, and output size limits.
- Runtime artifacts must be inspectable and stable enough for future reducers and skill wrappers.
- Existing static YAML workflows must remain compatible.
- Any app-thread dynamic support in this slice must stay read-only and follow the fixed JSON probe rule.

Verification:
- `git diff --check`
- `npx vitest run tests/dynamic-workflow*.test.ts`
- `npm run check`
- `bash scripts/smoke-cli.sh`
- Source audit that workflow scripts cannot access forbidden Node.js capabilities.
- Malicious fixture coverage for `fs`, `child_process`, `process.env`, `fetch`, `require`, dynamic import, `globalThis`, constructor escape attempts, prototype escape attempts, direct artifact writes, and target repo filesystem access.
- Worker sandbox fixture proving read-only target mutation is detected and produces a failed run, not PASS.
- Permission inheritance fixture proving `inherit-session` is rejected when parent permission cap is unknown or read-only.
- Trusted local fixture proving `inherit-session` records generated-current-session origin, script SHA-256, inherited sandbox metadata, and permits declared Codex worker file changes only when the parent session is write-capable.
- Untrusted workflow fixture proving copied-local, remote, registry, package, unknown-origin, and hash-mismatched workflows cannot request `inherit-session`.
- Preview fixture proving broad inherited permissions cannot skip explicit approval.
- Origin enum fixture proving unknown/unhandled origin values fail closed and `local-trust-record` remains disabled until implemented.
- App-thread fallback fixture proving unavailable write inheritance records `inherit-session-degraded-to-read-only`, requires explicit fallback approval, and appears in status/final summary.
- Artifact-boundary fixture proving inherited workers do not create out-of-run artifacts beyond declared target changes.
- Fixture proving preview writes script/capability/budget artifacts before any agent starts.
- Fixture proving reject-before-execute starts no agents and leaves target unchanged.
- Fixture proving `cwf.map` respects concurrency.
- Fixture proving max agent or timeout caps stop runaway dynamic workflows with a capped/partial result, not PASS.
- Fixture proving existing YAML workflows still pass registry and validation smoke.
- Controlled local smoke after Ender GO: run a simple dynamic JS workflow against a disposable local git repo, approve it, spawn at least two read-only workers, write artifacts, and produce a final report. If Ender does not approve real-smoke, stop at fixture/local evidence and report real-smoke not run.
- Reasonix/DeepSeek final review of implementation diff and docs; apply blocker/high findings before finalizing.
- GitHub CI green after push if pushed.

Iteration policy:
- Start by auditing current CLI, workflow loader, run-store, phase-engine, worker adapter, app-thread probe, and v1.10 safe-write boundaries.
- Implement one vertical slice at a time: parser/metadata -> preview artifacts -> constrained runtime -> agent.run -> map/concurrency -> CLI smoke -> docs.
- Run narrow tests after each slice before broadening.
- Keep updates in plain Chinese: what this slice lets CWF do, what it still refuses, and what evidence passed.
- Do not drift into v1.13 safe writes or v1.16 UX polish during v1.11.
- If a validation failure repeats twice, pause and inspect root cause before retrying.

Hard stops:
- Stop before allowing unrestricted Node.js APIs or shell access.
- Stop before using `vm` as the only security boundary.
- Stop before running dynamic workflows on a Node runtime that cannot enforce the required permission boundary.
- Stop before allowing dynamic agents to exceed the parent Codex session permission cap.
- Stop before allowing untrusted workflows to request `inherit-session`.
- Stop before allowing `inherit-session` without generated-current-session origin or another reviewed local trust record.
- Stop before allowing broad inherited permissions without non-skippable preview.
- Stop before silently degrading app-thread inherit-session behavior.
- Stop before executing remote JavaScript.
- Stop before enabling write operations from JavaScript.
- Stop before touching production, credentials, deploys, databases, payments, permissions, or external messages.
- Stop if dynamic workflow execution cannot be constrained enough to prevent direct filesystem/process/network access.
- Stop if existing YAML workflows regress.
- Stop after three repeated no-progress attempts on the same blocker and report the exact blocker.

Pause if:
- Pause and ask Ender before running controlled local smoke that starts real Codex workers.
- Pause and ask Ender if a useful v1.11 capability would require changing the public safety boundary.
- Pause and report exact evidence if Reasonix raises blocker/high findings after implementation.

Stop when:
- Stop when v1.11 acceptance is implemented, tests/smoke pass, Reasonix review is handled, docs are updated, and commit/push/CI status is reported.

Final response:
- Explain in human terms what JS dynamic workflows can now do and what they still refuse to do.
- List files changed, verification commands, live/fixture evidence, Reasonix review result, commit hash, push status, and CI status if pushed.
```
